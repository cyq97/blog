# GUIX 入门笔记



GUIX 拥有一个执行大部分 GUI 处理工作的内部线程。该线程由应用软件通过调用 ***gx_system_initialize***，随后调用 ***gx_system_start*** 来创建。

GUIX 内部线程执行循环由**三个操作**组成。

1. 首先，GUIX **从 GUIX 事件队列中获取事件**，并将这些事件分发至 GUIX 窗口和控件进行处理。

   事件通常由周期性定时器、触摸屏或键盘等输入设备，以及 GUIX 控件自身在处理用户输入时推送至 GUIX 事件队列。

   事件沿控件链向上传递的过程会一直持续，直到事件被处理完毕，或事件传递至根窗口，此时事件将被丢弃。

2. 接着，在所有事件处理完毕后，GUIX **判断是否需要刷新屏幕**，

   若需要则执行更新显示图形数据所需的处理，主要通过调用被标记为脏的窗口和控件的绘制函数来实现。

3. 最后，GUIX 挂起自身线程，**直至**有新的一个或多个输入**事件到达**。

内部 GUIX 线程的**优先级**由 `#define GX_SYSTEM_THREAD_PRIORITY` 定义。该值**默认是 16**（中等优先级），但可通过在 gx_port.h 或 gx_user.h 头文件中指定该值来修改，从而覆盖默认值。

GUIX线程**时间片**类似地由 `#define GX_SYSTEM_THREAD_TIMESLICE` 定义，**默认值为10 ms**。

系统线程的**栈大小**由`#define GX_THREAD_STACK_SIZE`定义，该定义位于***gx_port.h***头文件中，但也可通过在 gx_user.h 头文件中指定该值来重写。



GUIX 系统组件默认配置为利用 ThreadX 实时操作系统来提供线程服务、事件队列服务和定时器服务等功能。通过使用预处理指令 **GX_DISABLE_THREADX_BINDING** 并重新构建 GUIX 库，GUIX 可轻松移植到其他操作系统。这会从 GUIX 源代码中移除对 ThreadX 的依赖，并允许应用程序开发者基于目标系统提供的任意实时操作系统（RTOS）实现所需的操作系统服务。附录 F - GUIX 实时操作系统绑定服务 介绍了将 GUIX 移植到 ThreadX 以外的操作系统时需要实现的各类服务。





GUIX 为每个控件和画布实现了 ***脏区域***  这一概念，脏区域指的是需要重新绘制的区域。**控件只能向先前标记为脏的区域进行绘制**。当控件的绘制函数被调用时，所有绘制操作会在内部被裁剪到先前定义的脏矩形范围内，在该区域外进行的绘制操作会被忽略。

部件和窗口通过调用 API 函数 ***gx_system_dirty_mark*** 将自身标记为脏。该函数会**将整个部件或窗口标记为需要重绘**。另一个可替代的函数是 ***gx_system_dirty_partial_add***，调用它可以仅将窗口或部件的**某一部分标记为脏**。

将小部件标记为脏状态，然后仅在所有输入事件处理完毕后才重绘这些小部件的这种模型被称为 ***延迟绘制 (deferred drawing)***。GUIX 的**延迟绘制算法和脏列表维护机制旨在提高绘制效率**。由于绘制操作通常开销较大，GUIX 会尽力避免不必要的绘制操作。

GUIX 维护着一个脏部件列表，即那些可见且因状态变更或新变为可见而需要重绘的部件。该脏列表可让 GUIX 仅执行一次画布刷新操作，以反映 UI 状态的所有当前变更，而非在每次 UI 变更时都进行画布刷新，从而提升绘制性能。此脏列表同样由 GUIX 系统组件维护。







绘图操作在 GUIX 的 ***画布*** 上进行。画布是一块用于存储图形数据的内存区域。**画布是否直接与硬件帧缓冲区相连，取决于系统架构和内存限制**。

在进行任何绘图操作前，必须先调用 ***gx_canvas_drawing_initiate*** 应用程序接口函数来打开画布以进行绘图。该接口会为画布做好绘图准备，并建立当前的 *绘图上下文*。当 GUIX 执行系统画布刷新时，会在调用控件级绘图接口之前打开画布进行绘图并建立绘图上下文。因此，控件无需在自身的绘图函数中主动发起画布上的绘图操作

但是，如果应用程序希望在标准 GUIX 延迟绘图算法流程之外，立即在画布上执行绘图，则必须在调用任何其他绘图 API 函数之前直接调用***gx_canvas_drawing_initiate***，并且在立即绘图完成后必须调用***gx_canvas_drawing_complete***。





***模态对话框执行 (Modal dialog execution)*** 指的是向用户呈现一个窗口，该窗口必须以某种方式关闭后，其他 GUIX 窗口或部件才能接收用户输入。在对话框窗口显示期间，模态对话框会捕获所有用户输入，无论触摸或鼠标输入事件的 x、y 坐标位置如何。

模态对话框由应用软件触发，具体步骤为：首先通过调用***gx_window_create***以常规方式创建窗口，然后调用 GUIX API 函数***gx_window_execute***。

调用 ***gx_window_execute*** 函数时，GUIX 会进入本地事件处理循环。***gx_window_execute*** 函数不会返回给调用方，直到对话框窗口通过用户输入或调用 ***gx_window_close*** 关闭。因此，**切勿从 GUIX 内部线程以外的任何线程调用 *gx_window_execute* 函数**，这一点至关重要。



为了提供显示效果、动画以及对应用程序周期性请求的支持，GUIX 使用一个 ThreadX **定时器**。该单个定时器用于满足 GUIX 所有与时间相关的需求。**默认**情况下，GUIX 内部定时器的处理频率**通过常量 `GX_SYSTEM_TIMER_MS` 设为 20 毫秒**，该常量在 `gx_api.h` 中定义，除非该常量先前已在 `gx_port.h` 或 `gx_user.h` 头文件中定义。应用程序可在构建 GUIX 库时通过编译选项更改默认频率，或**在 *gx_user.h* 中重新定义它**。





GUIX 为每种底层绘图函数在每种支持的颜色深度和格式下都提供了默认软件实现。这些函数通过 **GX_DISPLAY** 结构体中维护的函数指针来调用。在创建特定硬件的驱动程序时，驱动程序通常会用针对目标硬件的函数覆盖其中一部分函数指针。

**典型的硬件显示驱动程序的实现方式**是：

1. 首先为所需的**色深**和**格式**创建**默认的 GUIX 显示驱动程序**，
2. 然后硬件驱动程序会**替换**那些需要针对特定硬件实现进行**优化或自定义的函数**。

GUIX 支持从 1 位每像素单色格式到 32 位带透明度的红-绿-蓝格式等多种像素颜色格式。GUIX 还支持每一大颜色深度类别内的多种变体，例如红-绿-蓝与蓝-绿-红字节顺序、打包像素与字对齐像素格式以及阿尔法通道。目前支持 25 种不同的颜色格式，且随着硬件厂商推出新的变体，该列表还在不断增加。

需要运行时内存分配和释放服务的 GUIX 服务包括：

- 从外部存储将二进制资源加载到 GUIX 运行时环境中。
- 软件运行时 JPEG 图像解码器
- 软件运行时 PNG 图像解码器
- 将文本小部件与 GX_STYLE_TEXT_COPY 配合使用。
- 运行时像素图缩放与旋转工具函数
- 运行时屏幕与控件块分配

运行时 JPEG 和 PNG 解码器为可选组件。大多数 GUIX 应用程序允许使用 GUIX Studio 工具对所有所需图像文件进行**预解码**，并将其存储为专用的 GUIX 像素图数据资源。





**GUIX 系统的基本组件**

1. GX_SYSTEM

   GUIX 系统组件，负责初始化、事件、计时器、字符串表以及可见控件层次结构的管理。

2. GX_CANVAS

   绘图区域。Canvas 可以是硬件帧缓冲区的轻量级抽象，也可以是纯内存画布。画布的类型由传递给 gx_canvas_create API 函数的参数决定。

3. GX_CONTEXT

   绘图上下文组件。绘图上下文包含有关屏幕、画布、画笔以及当前绘图操作的裁剪区域的信息。

4. GX_DISPLAY

   提供 API 和驱动级实现，使你的应用程序和 GUIX 小部件能够在画布上执行绘制。GX_DISPLAY 经过实现，可使用画布所需的颜色格式在每个画布上正确渲染图形。GX_DISPLAY 组件还会管理可用于绘制到与每个显示器关联的画布的小部件的资源（颜色、字体和像素图）。

5. GX_WIDGET

   基础可见小部件对象及相关API。所有GUIX小部件类型均基于或派生自基础GX_WIDGET类型。

6. GX_UTILITY

   该组包含用于处理矩形的实用函数、字符串转换函数以及非 ANSI 数学函数。



GX_SYSTEM

GUIX 系统组件提供与 GUIX **屏幕栈**相关的服务，这是一项可选功能，支持创建虚拟控件栈。应用程序可在运行时将屏幕**压入、弹出**该栈并从中获取屏幕。屏幕栈在**管理复杂菜单系统时十分实用**，因为用户进入菜单系统中不同状态的路径多种多样。要返回菜单系统中的上一状态，只需先压入前一屏幕状态，再显示新屏幕；当当前屏幕关闭时，允许新屏幕从屏幕栈中弹出前一状态即可。



GX_CANVAS

画布组件负责所有与画布相关的处理。画布实际上是一个虚拟帧缓冲区。你的应用程序必须创建至少一个画布才能生成图形输出。通常，对于系统支持的每一个物理显示器，你都需要创建至少一个画布。

所有 GUIX 绘图都在画布上进行。在配置较简单或内存受限的系统中，可能只有一个画布，它可能直接与可见的帧缓冲区相连；而内存更大、图形需求更高级的系统则可能需要多个画布。为单个显示器提供**多个画布可实现屏幕和窗口淡入淡出**等效果。画布主要分为两种类型：简单画布和托管画布。



GX_DISPLAY

显示组件是GUIX中的核心组件，因为它负责处理所有显示对象，而这些显示对象本身包含一个或多个画布、部件和窗口。显示组件还通过一系列函数指针与每个显示对应的底层硬件屏幕驱动程序进行交互。
